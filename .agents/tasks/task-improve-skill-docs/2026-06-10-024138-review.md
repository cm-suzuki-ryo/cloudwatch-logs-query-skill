# Documentation hardening for AI query generation guardrails

This change adds two substantial reference sections to the CloudWatch Logs query builder skill: a formal pipeline ordering specification with an ASCII flow diagram and constraint table, and a catalog of seven AI-specific anti-patterns with mistake/fix pairs. A minor metadata addition ("Last verified: 2026-06") lands on the reference doc. The goal is to reduce incorrect query generation by giving AI models explicit ordering rules and named failure modes to match against before emitting output.

Watch for: redundant coverage between the new "Command Pipeline Ordering Rules" section and the pre-existing "Command Ordering & Gotchas" section (confirmed); the pipeline flow diagram omits commands that appear elsewhere in the document — `dedup`, `pattern`, `diff`, `join`, `relevantfields` — leaving the "valid pipeline" incomplete (confirmed); anti-pattern #1's "Correct approach" doesn't actually solve the stated problem of post-aggregation filtering (confirmed).

## High-level view

The pipeline ordering section formalizes rules previously scattered across tips and gotchas into a single scannable structure. However, the diagram presents a simplified four-stage pipeline that doesn't account for several commands documented elsewhere in the same file, and the section sits directly above an older section covering much of the same ground without any cross-reference or deduplication.

The anti-patterns section catalogs seven concrete mistakes with example/explanation/fix triples — each maps to a real failure mode that language models hit. One correctness issue: anti-pattern #1 claims to fix "filtering after stats" but the corrected query just sorts, leaving the reader without a real solution for the use case (wanting only groups with count > N).

The actual fix for "show only services with more than 10 errors" is a chained `stats` with a `filter`-like approach (e.g., omitting low-count rows via a second aggregation) or restructuring the query. ## Structural redundancy between ordering sections

The skill file now has two sections covering command ordering: "Command Pipeline Ordering Rules" (new) and "Command Ordering & Gotchas (verified)" (pre-existing). Three constraints appear in both:

- `dedup` cannot be followed by `sort` — stated in the constraints table AND in the gotchas bullet list
- `diff` only after `pattern` — same
- `jsonParse` dot access requires a separate `fields` command — not in the ordering table but covered by anti-pattern #7 AND the gotchas list

For an AI model consuming this as a skill file, duplicate rules with slightly different phrasing ("syntax error" vs "correct order: `sort` then `dedup`") create ambiguity about which is authoritative. The simplest fix is to collapse "Command Ordering & Gotchas" into the new section, or reduce the gotchas section to only items that don't fit the pipeline model (like `relevantfields` quirks and `addtotals` behavior).

## Pipeline diagram completeness

The ASCII diagram shows a clean four-stage flow:

```
filterIndex → fields/parse/filter → stats → sort/limit/display
```

Commands missing from this picture:

| Command | Where it fits | Documented elsewhere in this file |
|---------|---------------|-----------------------------------|
| `dedup` | After sort, before or after filter (complex rules) | Yes — gotchas section |
| `pattern` | Pre-aggregation (alternative to stats) | Yes — operational patterns |
| `diff` | Only after pattern | Yes — gotchas |
| `join` | Mid-pipeline | Yes — cross log group section |
| `relevantfields` | Pre-aggregation with restrictions | Yes — gotchas |
| `addtotals` | After fields | Yes — gotchas |

The diagram would benefit from either expanding to show these, or adding a note: "Diagram shows the core aggregation pipeline. See Ordering Constraints table and Command Ordering & Gotchas for additional commands."

## Anti-pattern #1: filter-after-stats fix is incomplete

The stated mistake is:

```
stats count(*) as errors by service
| filter errors > 10
```

The "Correct approach" offers:

```
stats count(*) as errors by service
| sort errors desc
```

Sorting doesn't solve the original intent (threshold filtering on aggregated values). The guidance says "Apply filtering conditions before stats, or use a chained stats to narrow results" but doesn't show the chained-stats approach. Since this is the #1 most common AI generation error, the correct approach should demonstrate the actual workaround — or explicitly acknowledge that Logs Insights QL has no `HAVING` equivalent and the standard workaround is to sort descending and use `limit`, accepting that low-count groups will be at the bottom rather than excluded.

## Anti-pattern #6: vague corrective guidance

Anti-pattern #6 (unnest on split) correctly identifies the `MalformedQueryException` but the "Correct approach" is vague ("use multiple parse commands or filter with like/regex"). A single concrete example would strengthen it, since the whole point of the anti-patterns section is showing the fix, not just naming it.
